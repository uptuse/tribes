# Manus Feedback — Round 25: Custom Maps + Replay Playback (SONNET)

**MODEL:** **SONNET 4.6** (1M context)
**Severity:** Standard
**Type:** Implementation — extend the single-map prototype to a custom-map system and add full replay-playback functionality on top of the R23 capture-only scaffold

---

## 1. Context

The build has shipped a complete multiplayer experience through R24 (matchmaking, telemetry, voice, classes, settings v2). Two pieces of player-facing content remain that determine whether the game has long-term replay value: the ability to play maps other than the default Raindance-clone, and the ability to watch a recorded match.

R25 delivers both: a JSON-defined map format with terrain heightmap, building placement, flag/spawn locations, and gameplay constants; a map browser UI; and full replay playback with scrub controls operating on the binary buffers captured in R23.

---

## 2. Concrete tasks

### 2.1 P0 — Map format + 3 launch maps (~50 min)

Define a JSON map format in `client/maps/schema.md`. Each map ships as a `.tribes-map` file (gzipped JSON) with metadata (name, author, max players, recommended class mix) plus terrain (heightmap as base64-encoded Float32Array of 256×256 elevation values), structures (array of `{type, position, rotation, scale}` for buildings/towers/turrets), gameplay points (red flag, blue flag, spawn arrays per team, inventory station positions), and atmosphere (sky color, sun angle, fog distance, ambient light intensity).

Server in `sim.ts` adds `loadMap(mapJson)` that initializes the simulation state from the map data instead of the hardcoded constants. Client renderer in `renderer.js` reads the same JSON and constructs the Three.js scene from it on match start.

Three launch maps ship in `client/maps/`: `raindance.tribes-map` (port of the current default), `dangercrossing.tribes-map` (smaller arena-style map for fast matches), `iceridge.tribes-map` (large open snow map favoring jet-and-ski gameplay). Each map's heightmap is procedurally generated by a small Bun script `tools/genmap.ts` that takes parameters and writes the gzipped output. Maps total <500KB compressed combined.

### 2.2 P0 — Map browser UI in lobby (~20 min)

The Create Custom Lobby flow extends with a map picker. Players see a 3-card carousel of the launch maps with thumbnail (rendered offscreen at lobby creation by Three.js + saved to localStorage as data URL), name, author, recommended size, and a "Play this map" button. The selected map ID flows to the server in the createLobby request.

The lobby browser endpoint extends each entry with the map name and a 64×64 thumbnail. The Quick Match algorithm respects map preference if set (player can mark favorite maps in settings).

### 2.3 P0 — Replay playback engine (~50 min)

The R23 capture writes per-tick snapshots to a binary buffer. Build the playback side now. New file `client/replay.js` (~300L target).

Replay loader: file picker accepts `.tribes-replay` files, parses the header (match metadata, tick count, player roster with classes), and scrubs through the snapshot stream. Provides three control modes: realtime (1.0× speed, default), scrub (slider drags to absolute tick), and step (frame-by-frame for analysis).

Renderer integration: when in replay mode, the C++ WASM physics tick is bypassed; the renderer reads positions/rotations directly from the decoded snapshot. Camera is detached from the local player and supports three view modes: follow-player (cycle players with arrow keys), free-cam (WASD + mouse to fly), and overhead (top-down strategic view zoomed to fit map bounds).

UI overlay: bottom timeline shows the match's significant events (kills, captures, score changes) as ticked markers. Click an event to jump to that tick. Play/pause/speed-control (0.25×, 0.5×, 1×, 2×, 4×) buttons. Esc returns to main menu.

Server replay download: the existing `GET /replay/:matchId` endpoint from R23 streams the binary file. The match-end overlay's "Save Replay" button works end-to-end. Players can also paste a replay URL into the main menu to download and view.

### 2.4 P1 — Map editor (basic) (~40 min)

Players can create custom maps in-browser via a basic editor accessed from the main menu's "MAP EDITOR" button. The editor presents a top-down map view of a 256×256 heightmap and provides three tools: brush (click-drag raises/lowers terrain with falloff curve, modifier key inverts), structure placement (click drops a building/tower/turret with rotation via mouse wheel), and gameplay-point placement (click drops red flag, blue flag, or spawn point).

Save: the editor exports a `.tribes-map` file via the browser's download API. Load: the editor accepts an existing `.tribes-map` for further editing. The editor's Test button launches a local-only single-player match on the current map state for quick iteration.

The editor is intentionally minimal in R25 — no texture painting, no advanced terrain sculpting, no scripting. R26+ will iterate based on player demand and editor playtime telemetry.

### 2.5 P1 — Replay sharing (~15 min)

The match-end overlay's "Save Replay" extends with a "Share Replay" button. On click, the client uploads the replay file to a server endpoint `POST /replay-upload`, which stores it on Cloudflare R2 (object storage; ~$0.015 per GB-month, free tier 10GB) and returns a short shareable URL. Players paste the URL into the main menu to play the replay. Lifetime: 30 days, then auto-delete.

### 2.6 P1 — In-match map vote (~15 min)

The match-end overlay extends the existing 75% play-again vote to support a map-vote variant. When initiated by the host or by majority vote during the match-end screen, a 3-option map shuffle appears (current + 2 random unplayed maps from the rotation). Players click their preference; the most-voted wins for the next match. If tied, current map wins.

### 2.7 P2 — Map preview rendering (~15 min)

When a player hovers over a lobby in the lobby browser, the lobby's map thumbnail enlarges and a 5-second autoplay loop of the map's flyover renders (precomputed at map publish time and stored as a 256×256 H.264 mp4 looped via `<video>` tag). This sells the map's vibe before the player commits. Storage cost negligible (each clip ~200KB).

### 2.8 P3 — Replay annotation (~10 min)

While viewing a replay, players can press `M` at any tick to drop a marker with a text note (e.g., "great cap here"). Markers persist in the replay file's metadata (small append at end). When the replay is re-shared, viewers see other players' markers. Markers display on the timeline as small flags.

## 3. Acceptance criteria (must hit 6 of 8)

The first six lines below capture what must function end-to-end. The last two are nice-to-have polish.

A `.tribes-map` JSON file loads on the server and produces a playable match with the documented terrain, structures, and gameplay points. Three launch maps work and feel distinct in self-play. The lobby browser shows map thumbnails and the Create Custom flow includes a map picker. A `.tribes-replay` file loaded into the client plays back the match with all players at correct positions, weapons firing, and score updating in sync. Replay playback supports realtime, scrub, and step modes with a working timeline of kill/capture markers. The basic map editor allows brush-tool terrain editing and structure placement, then exports a valid `.tribes-map`. Map preview videos auto-play on lobby hover. Replay annotation markers persist and display correctly.

## 4. Compile/grep guardrails

Standard guardrails: no `EM_ASM \$1[6-9]`, all new server files in `server/*.ts` typed without `:any`, all new client files as ES modules, dependencies pinned, vanilla-JS client, `bun build` and `bun run test` clean. R2 client should use the standard AWS S3 SDK pinned version (R2 is S3-compatible).

---

## 5. Time budget

A full Sonnet round at 180-220 minutes covers all P0 and P1 items. The replay playback engine is the largest single piece because it requires fully detaching the renderer from live WASM input. The map format and three launch maps depend on the simulation refactor in `sim.ts:loadMap` which has moderate ripple. Map editor is mostly UI and can be aggressive in scope reduction if budget runs short.

---

## 6. Decision authority for ambiguities

If the heightmap encoding (base64 Float32 256×256 = ~340KB raw) inflates map size beyond targets, swap to delta-encoded Int16 quantized to 0.1m precision (~130KB raw, ~30KB gzipped); document the precision trade-off in `client/maps/schema.md`. If R2 setup adds friction beyond 15 min of work, ship Save Replay locally only and defer Share Replay to R26 (note in `open_issues.md`). If the map editor runs into performance issues at 256×256 brush operations, downsample the working buffer to 128×128 and upsample on save with bilinear interpolation. If replay playback struggles to maintain 60 FPS at 4× speed, cap the speed to 2× and document. If structure placement in the editor accidentally allows out-of-bounds spawns, add a 10m boundary clamp.
